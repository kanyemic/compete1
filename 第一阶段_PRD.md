# 医影挑战赛第一阶段 PRD

## 1. 文档目标

本文档用于定义医影挑战赛第一阶段的产品范围，目标是在尽量少改动方向的前提下，把当前前端原型升级为一个可以上线验证留存的 MVP。

## 1.1 当前实现快照（更新于 2026-04-12）

当前版本已经实现这些第一阶段核心体验：

- 游客身份自动生成与本地持久化
- 单人连胜、本地历史统计、结果页
- 每日挑战固定题组、结果页、今日榜入口
- 排行榜双榜切换与当前用户成绩卡片
- 错题本详情与基础筛选
- 个人页统计、最近记录、近 7 天活跃情况
- 首页回访提示与最近成绩摘要

当前仍未完全达到 PRD 目标的部分：

- 基础题库数量仍不足 30-50 题
- 题目来源 / 审核人 / 审核流程展示还不完整
- 埋点当前为本地事件流，尚未接入真实分析平台
- 排行榜“当前用户真实名次”在 Supabase 环境下仍需继续增强

第一阶段不追求做全功能平台，只验证三件事：

1. 用户是否愿意持续刷题和回访。
2. 每日挑战是否能建立日活心智。
3. 排行榜、历史记录和错题机制是否能带来留存提升。

## 2. 产品目标

### 业务目标

- 让用户在首次进入后 3 分钟内完成至少 1 局挑战。
- 让用户形成“明天还要回来做每日挑战”的预期。
- 为第二阶段会员化和正式排位提供用户行为数据基础。

### 用户目标

- 快速进入练习状态，不需要复杂学习成本。
- 能看到自己的成绩、历史和进步。
- 能清楚知道自己错在哪里、应该补什么。

## 3. 目标用户

第一阶段聚焦这三类用户：

- 医学生
- 规培生 / 住院医
- 影像、病理、急诊等相关科室的年轻医生

不优先覆盖：

- 纯 C 端泛娱乐用户
- 深度机构协作用户
- 专家内容创作者

## 4. 第一阶段范围

## 4.1 包含功能

### 4.1.1 登录与基础用户记录

目标：让成绩、历史、排行榜有归属。

能力要求：

- 支持最简登录方式
- 首次进入可自动生成游客身份
- 登录后保留昵称、头像、基础统计
- 用户成绩与挑战记录可持久化

### 4.1.2 单人连胜升级版

目标：保留当前最直接的爽点玩法。

能力要求：

- 支持继续当前无限连胜模式
- 展示本次连胜、历史最佳、总答题数
- 结束后展示错题原因和再来一局入口
- 成绩写入个人历史记录

### 4.1.3 每日挑战

目标：建立回访理由。

能力要求：

- 每天固定一组题目，所有用户一致
- 每位用户每天只能生成一次正式成绩
- 支持展示今日分数、耗时、今日排名
- 第二天自动刷新新的挑战题组

### 4.1.4 排行榜

目标：让成绩具有社会参照。

能力要求：

- 至少包含两个榜单：最高连胜榜、每日挑战榜
- 支持查看自己的名次和分数
- 支持展示 Top 10 或 Top 50
- 排行榜数据来自真实记录而非 mock

### 4.1.5 错题本

目标：把一次挑战变成学习资产。

能力要求：

- 自动记录答错题目
- 保存题目、正确答案、用户答案、解析、时间
- 支持再次查看错题详情
- 支持按专科或难度做基础筛选

### 4.1.6 历史记录 / 个人主页基础版

目标：沉淀个人训练轨迹。

能力要求：

- 查看最近挑战记录
- 查看历史最佳连胜
- 查看累计答题数、正确率
- 查看最近 7 天活跃情况

### 4.1.7 基础题库扩充

目标：避免内容过少导致早期流失。

能力要求：

- 基础病例至少扩展到 30-50 题
- 按专科做最小分类，例如胸片、头颅 CT、骨科、病理
- 每道题包含图像、描述、选项、答案、解析、难度
- 所有内容在第一阶段先采用人工整理或可信公开资料

## 4.2 不包含功能

以下能力明确放到第二阶段或更后面：

- 会员体系
- 支付、商城、虚拟货币
- 真人实时匹配
- 房间对战 / 教学房
- UGC 题库创建
- AI 自动生成并直接上线题目
- B 端机构管理后台

## 5. 核心用户流程

### 流程 A：首次用户

1. 用户进入首页。
2. 看到单人连胜、每日挑战、排行榜入口。
3. 进入单人连胜完成至少 1 局。
4. 对局结束后看到成绩与解析。
5. 系统提示保存成绩并查看历史记录。
6. 用户返回首页后看到今日挑战入口。

### 流程 B：回访用户

1. 用户再次进入产品。
2. 首页突出展示“今日挑战已更新”。
3. 用户完成每日挑战。
4. 用户查看今日榜单和个人排名。
5. 用户进入错题本查看失误原因。

### 流程 C：学习型用户

1. 用户刷单人连胜。
2. 结束后进入错题本。
3. 查看解析并再次练习。
4. 在个人页看到正确率和历史趋势。

## 6. 页面与模块需求

### 6.1 首页

必须展示：

- 品牌标题
- 单人连胜入口
- 每日挑战入口
- 排行榜入口
- 个人记录快捷入口

推荐展示：

- 今日挑战剩余时间
- 用户历史最佳
- 最近一次成绩摘要

### 6.2 对局页

必须展示：

- 病例图像
- 病例描述
- 选项
- 倒计时
- 当前得分或当前连胜

结算时必须展示：

- 正确答案
- 用户是否答对
- 题目解析
- 下一题或返回入口

### 6.3 每日挑战页

必须展示：

- 今日挑战标题与日期
- 当前题号 / 总题数
- 累计得分
- 完成后结果页
- 今日排名入口

### 6.4 排行榜页

必须展示：

- 切换榜单
- 名次、昵称、分数
- 当前用户名次

### 6.5 错题本页

必须展示：

- 错题列表
- 错题详情
- 正确答案与解析
- 筛选项

### 6.6 个人页

必须展示：

- 昵称头像
- 历史最佳
- 累计答题数
- 正确率
- 最近记录

## 7. 数据需求

第一阶段至少要有这些核心数据对象：

- User
- QuestionCase
- DailyChallenge
- DailyChallengeAttempt
- SoloRun
- WrongQuestionRecord
- LeaderboardSnapshot 或实时排行榜聚合数据

关键字段建议：

- 用户：id、昵称、头像、创建时间
- 题目：专科、难度、图片、题干、选项、答案、解析、来源
- 挑战记录：模式、得分、耗时、是否完成、时间戳
- 错题记录：用户、题目、错误答案、答题时间

## 8. 埋点与指标

第一阶段必须埋点：

- 首页曝光
- 首页点击玩法入口
- 开始对局
- 提交答案
- 答题正确/错误
- 完成单人连胜
- 完成每日挑战
- 查看排行榜
- 查看错题本

重点指标：

- 新用户首局完成率
- 首日平均答题数
- 每日挑战参与率
- 次日留存
- 7 日留存
- 错题本访问率
- 排行榜访问率

## 9. 验收标准

第一阶段完成的标准：

- 用户可以登录或游客进入并保存成绩
- 单人连胜使用真实持久化成绩
- 每日挑战可按天切换并生成真实榜单
- 排行榜不再依赖 mock 数据
- 错题本和个人历史可访问
- 基础题库数量达到上线最小规模

## 10. 风险与约束

- 医学内容必须明确用于教育训练，不能暗示临床诊断用途。
- 第一阶段题库不要完全依赖 AI 自动生成，避免准确性问题。
- 如果资源有限，优先保证题库质量和记录系统，再优化视觉和动画。
- 真人 PvP 不是第一阶段必需项，不应抢占每日挑战和错题本的资源。

## 11. 第一阶段上线结论

第一阶段的正确策略不是“继续堆玩法”，而是把当前原型做成一个真正可记录、可复盘、可回访的训练产品。

只要这几个点成立：

- 用户能顺畅开局
- 每天有固定挑战
- 成绩和错题能沉淀
- 用户能看到自己和别人差距

这个项目就具备进入第二阶段商业化验证的基础。
