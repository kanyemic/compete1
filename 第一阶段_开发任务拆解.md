# 医影挑战赛第一阶段开发任务拆解

## 1. 目标

本拆解用于把第一阶段 PRD 变成可执行任务，默认按 `前端 / 后端 / 数据 / 内容` 四条线推进。

建议开发顺序：

1. 先打通数据模型和持久化。
2. 再接入首页与玩法入口。
3. 然后补每日挑战、排行榜、错题本。
4. 最后做个人页和体验优化。

## 3. 任务拆解

### 3.1 数据与后端

#### 任务 1：定义核心数据模型

输出：

- 用户表
- 题目表
- 每日挑战表
- 每日挑战成绩表
- 单人连胜记录表
- 错题记录表

验收：

- 能覆盖第一阶段所有页面数据来源
- 字段能支持排行榜和历史记录聚合

#### 任务 2：落地题库数据源

输出：

- 将当前 mock 题库迁移为结构化数据源
- 支持按专科、难度查询题目

验收：

- 前端不再依赖内嵌数组作为唯一题源
- 支持最少 30-50 道题目

#### 任务 3：实现每日挑战出题逻辑

输出：

- 按日期生成每日题组
- 所有用户当日共享同一题组
- 支持防重复提交正式成绩

验收：

- 同一天内同一用户只能有一条正式成绩
- 第二天自动切换新题组

#### 任务 4：实现排行榜聚合接口

输出：

- 最高连胜榜接口
- 每日挑战榜接口
- 当前用户排名接口

验收：

- 前端可直接查询榜单
- 排名结果和历史成绩一致

#### 任务 5：实现错题记录接口

输出：

- 错题写入接口
- 错题列表接口
- 错题详情接口

验收：

- 用户答错后可在错题本看到记录

### 3.2 前端重构

#### 任务 6：抽离页面路由和状态结构

当前问题：

- [App.tsx](/Users/kanyemic/Desktop/medical challenge/compete1/App.tsx) 目前承载了几乎所有流程和视图，后续功能增加后会迅速变得难维护。

输出：

- 将首页、对局页、排行榜页、个人页、错题本页拆分为独立页面或模块
- 将游戏流程状态与用户数据状态分离

验收：

- 首页、玩法页、榜单页不再全堆在一个文件里

#### 任务 7：首页升级

输出：

- 增加每日挑战入口
- 增加个人记录入口
- 展示历史最佳和今日挑战状态

验收：

- 用户进首页后知道“今天该做什么”

#### 任务 8：单人连胜接真实数据

输出：

- 对局结束后保存成绩
- 读取历史最佳和累计统计

验收：

- 结束页显示真实历史数据

#### 任务 9：新增每日挑战页面

输出：

- 每日挑战流程页
- 每日挑战结果页
- 今日排名入口

验收：

- 用户可完整完成当天挑战并看到结果

#### 任务 10：排行榜页改造

当前问题：

- [components/Leaderboard.tsx](/Users/kanyemic/Desktop/medical challenge/compete1/components/Leaderboard.tsx) 当前仍依赖 mock 数据。

输出：

- 接真实接口
- 支持“最高连胜榜 / 每日挑战榜”切换
- 展示当前用户名次

验收：

- 页面数据刷新后稳定可用

#### 任务 11：新增错题本页面

输出：

- 错题列表页
- 错题详情弹层或详情页
- 基础筛选能力

验收：

- 用户至少能查看最近错题

#### 任务 12：新增个人页

输出：

- 历史最佳
- 累计答题数
- 正确率
- 最近记录

验收：

- 用户能感知自己的进步轨迹

### 3.3 登录与身份

#### 任务 13：接入最简身份体系

建议优先级：

- 首选游客 + 本地持久化
- 其次手机号或邮箱登录

输出：

- 首次进入自动生成用户身份
- 后续可绑定正式账号

验收：

- 用户刷新页面后成绩不丢

### 3.4 内容与题库

#### 任务 14：整理第一阶段基础题库

输出：

- 胸片题库
- 头颅 CT 题库
- 骨科题库
- 病理题库

验收：

- 每个分类至少有基础题量
- 所有题目有标准解析
- 图片来源和内容来源可追溯

#### 任务 15：建立题目审核表

输出：

- 题目来源
- 审核人
- 审核状态
- 更新时间

验收：

- 后续 AI 扩题或人工扩题时可复用

### 3.5 埋点与运营基础

#### 任务 16：接入埋点

输出：

- 首页点击
- 开局
- 提交答案
- 完成挑战
- 查看排行榜
- 查看错题本

验收：

- 可拉出基础漏斗

#### 任务 17：增加首页回访提示

输出：

- 今日挑战状态
- 最近成绩摘要
- 连续活跃提示

验收：

- 回访用户一进来就能知道下一步动作

## 4. 推荐排期

### 第 1 周

- 数据模型设计
- 题库数据整理
- 前端结构拆分

### 第 2 周

- 单人连胜接真实数据
- 用户身份接入
- 排行榜接口与页面改造

### 第 3 周

- 每日挑战后端逻辑
- 每日挑战前端页面
- 成绩写入与榜单联动

### 第 4 周

- 错题本
- 个人页
- 埋点与上线收尾

## 5. 优先级建议

P0：

- 用户身份
- 题库数据
- 单人连胜持久化
- 每日挑战
- 排行榜

P1：

- 错题本
- 个人页
- 埋点

P2：

- UI 精修
- 动效
- 更复杂筛选与统计

## 6. 技术建议

基于当前仓库体量，第一阶段建议：

- 前端继续使用现有 React + Vite 架构
- 题库和成绩数据尽快迁出前端 mock
- 如果想快速上线，可优先选择轻量后端方案，例如 Supabase / Firebase / 自建简单 API
- 题目图片建议独立托管，避免后续扩题后前端仓库过重

## 7. 第一阶段完成定义

当以下条件全部成立，可视为第一阶段完成：

- 新用户进入后能直接开玩
- 每天有固定的每日挑战
- 成绩、榜单、错题、历史都能持久化
- 基础题库足够支撑一周以上训练
- 可以开始真实收集留存数据

## 8. 下一步衔接

第一阶段完成后，第二阶段可以自然衔接：

- 会员体系
- AI 复盘
- 正式积分排位
- 房间模式
- 教师题库

所以第一阶段开发时，要尽量避免把数据结构写死在“只有单人模式”的假设上。
