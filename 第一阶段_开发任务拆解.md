# 医影挑战赛第一阶段开发任务拆解

## 1. 目标

本拆解用于把第一阶段 PRD 变成可执行任务，默认按 `前端 / 后端 / 数据 / 内容` 四条线推进。

## 1.1 当前进度快照（更新于 2026-04-12）

当前代码已经从纯原型推进到“可本地运行的 MVP 骨架”，并支持 `本地持久化优先 + Supabase 可选接入`。

已经完成或基本完成的能力：

- 游客身份自动生成与本地持久化
- 单人连胜成绩落本地统计，并尝试同步 Supabase
- 每日挑战固定题组、本地今日记录、结果页与榜单入口
- 排行榜双榜切换与“当前用户成绩/名次”卡片
- 错题本列表、详情、基础筛选
- 个人页基础统计、最近记录、近 7 天活跃情况
- 首页回访提示、最近成绩摘要、今日挑战状态
- 关键流程埋点骨架（本地存储 + Supabase 可选入库）
- `App.tsx` 状态编排重构，主流程拆入 hooks

仍需继续推进的重点：

- 题库扩充到 30-50 题，并补齐来源/审核信息
- 排行榜“当前用户真实名次 / 排名稳定性”在 Supabase 环境下继续做深
- 埋点事件表虽然已落 schema 与前端同步逻辑，但仍需补监控与分析视图
- 题目审核字段与内容管理流程继续补齐
- 每日挑战和题库内容运营化

建议开发顺序：

1. 先打通数据模型和持久化。
2. 再接入首页与玩法入口。
3. 然后补每日挑战、排行榜、错题本。
4. 最后做个人页和体验优化。

## 5. 优先级建议

P0：

- 题库数据扩充
- 排行榜真实名次
- 题目审核字段
- Supabase 环境联调

P1：

- 埋点真实上报
- 题目来源展示
- 错题复练能力

P2：

- UI 精修
- 动效
- 更复杂筛选与统计

## 6. 技术建议

基于当前仓库体量，第一阶段建议：

- 前端继续使用现有 React + Vite 架构
- 题库和成绩数据尽快迁出前端 mock
- 如果想快速上线，可优先选择轻量后端方案，例如 Supabase / Firebase / 自建简单 API
- 题目图片建议独立托管，避免后续扩题后前端仓库过重

## 7. 第一阶段完成定义

当以下条件全部成立，可视为第一阶段完成：

- 新用户进入后能直接开玩
- 每天有固定的每日挑战
- 成绩、榜单、错题、历史都能持久化
- 基础题库足够支撑一周以上训练
- 可以开始真实收集留存数据

当前判断：

- 前三条已基本达到
- 后两条仍未完全达到，主要缺口在题库量和真实埋点上报

## 8. 下一步衔接

第一阶段完成后，第二阶段可以自然衔接：

- 会员体系
- AI 复盘
- 正式积分排位
- 房间模式
- 教师题库

所以第一阶段开发时，要尽量避免把数据结构写死在“只有单人模式”的假设上。
